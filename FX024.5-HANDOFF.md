# FX024.5 Handoff - Beta Launch Hardening

## Status

FX024.5 is implemented. Fast local checks pass, but full DB-backed verification
is blocked on this machine because no Postgres server is reachable at
`localhost:5432` and Docker Desktop is not providing the local container.

## Sprint Direction

Make the platform safe to hand out beta invite links: durable cross-instance
rate limiting on sensitive routes, a single verify pipeline, and an E2E
golden-path smoke test. No changes to AMM math, settlement logic, or ledger
semantics.

## What Changed

### Durable rate limiting

- `lib/rate-limit-upstash.ts`: fixed-window adapter over the Upstash Redis
  REST API (`INCR` + `PEXPIRE NX` + `PTTL` in one pipeline). Works in Node and
  edge runtimes, fails open with a logged error when Redis is unreachable, and
  never logs the token.
- `lib/rate-limit-config.ts`: factory mirroring the NFL provider pattern.
  Selects Upstash when `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN`
  are set; otherwise falls back to the in-memory adapter with a one-time
  warning. Exposes `enforceRateLimit`, `getClientIp`, and `RATE_LIMITS`.
- `lib/rate-limit.ts`: exported `InMemoryRateLimitAdapter`, added
  `RateLimitError` (429, code `RATE_LIMITED`).
- `lib/api-response.ts`: maps `RateLimitError` to a 429 JSON response with
  `x-ratelimit-limit/remaining/reset` headers.
- Route wiring:
  - `POST /api/trades` — 30/min per user
  - `POST /api/auth/login` — 10/min per IP
  - `POST /api/auth/signup` — 10/min per IP
  - `POST /api/beta-events` — 60/min per user
- `middleware.ts`: comment documenting that its IP bucket is per-instance and
  coarse only; durable enforcement lives at the route level.
- `.env.example`: Upstash variables documented.

### Verify pipeline

- `scripts/verify.mjs` + `npm run verify`: lint → typecheck → `vitest run` →
  build, failing fast. Before the test step it probes the Postgres server from
  `DATABASE_URL` and, when unreachable, exits 1 with concrete fix steps
  (start Docker Desktop, `docker compose up -d`, migrate, seed) instead of a
  Prisma stack-trace wall. Tests are never silently skipped.

### E2E smoke test

- `playwright.e2e.config.ts` + `npm run test:e2e` (separate from the a11y
  config; dev server on port 3211).
- `tests/e2e/golden-path.spec.ts`: signs up a fresh user through the UI,
  completes onboarding, buys YES on the first OPEN seeded market, confirms the
  balance drop and the portfolio position, logs in as the seeded admin,
  settles the market YES via the settlement API with a CSRF token, and asserts
  the payout in both the trader balance and a `SETTLEMENT_PAYOUT` ledger row.
  Rerunnable: each run uses a fresh user and consumes the next OPEN market.

### Tests

- `tests/rate-limit.test.ts`: 11 tests — Upstash window math from PTTL, limit
  exceeded, fresh-window reset, fail-open on network error and non-2xx
  (asserting the token never appears in logs), factory selection for both env
  states, one-time fallback warning, `enforceRateLimit` throwing
  `RateLimitError`, 429 mapping with headers, client IP extraction, and the
  agreed limit values.

## Launch checklist item

Before inviting beta users, create a free Redis database at
https://console.upstash.com and set `UPSTASH_REDIS_REST_URL` and
`UPSTASH_REDIS_REST_TOKEN` in the Vercel project env. Without them the app
still runs, but rate limits are per-instance only.

## Verification

Verified on 2026-07-16:

- `npm run lint` passed.
- `npm run typecheck` passed.
- `npx vitest run tests/rate-limit.test.ts` passed (11 tests).
- `npm run build` passed.
- `npm run verify:fast` passed.
- `npm run db:check` exits 1 with hosted-Postgres and optional-Docker setup
  guidance when `DATABASE_URL` is unreachable.
- `npm run verify` correctly passes lint/typecheck and then exits with the
  actionable Postgres-unreachable message before DB-backed tests.

Blocked:

- `npm run test` full suite requires a reachable Postgres database.
- `npm run test:e2e` requires a migrated and seeded Postgres database.

The blocker is environment-only: no local Postgres service exists, port 5432 is
closed, and Docker Desktop's engine is not responding reliably enough to start
the `fantasyx-postgres` compose service.

## Verification Infrastructure Follow-Up

Added after the Docker Desktop blocker:

- `scripts/db-check.mjs`: reusable DATABASE_URL validator and TCP reachability
  check. It supports hosted dev Postgres, local Docker Postgres, and clear
  no-database guidance.
- `scripts/verify.mjs`: now has `--fast` and `--full` modes.
- `npm run verify:fast`: no-DB gate (lint, typecheck, rate-limit tests, build).
- `npm run verify` / `npm run verify:full`: strict gate (lint, typecheck,
  database preflight, full Vitest suite, build, E2E smoke). It never silently
  skips DB-backed tests.
- `npm run db:check`, `npm run db:migrate`, and `npm run db:prepare`.
- README and DEPLOYMENT now recommend a hosted disposable dev Postgres database
  when Docker Desktop is unreliable, with Docker as an optional fallback.
